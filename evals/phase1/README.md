# Phase 1.2A — Evaluation Foundation

## Overview

Phase 1.2A provides a **tenant-aware, deterministic** evaluation framework for the AI Server.
It evaluates both retrieval quality and answer quality using **programmatic graders only**
(no LLM judges, no pairwise comparison).

```
evals/phase1/
├── README.md                     # This file
├── golden_set.jsonl              # Eval cases (JSONL format)
└── (results.json)                # Legacy — see evals/outputs/phase1/ instead
```

## Golden Set Schema

Each line in `golden_set.jsonl` is a JSON object with the following fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `case_id` | `str` | ✅ | Unique identifier for the eval case |
| `tenant_id` | `str` | ✅ | Tenant to run against (**must not be empty**) |
| `module` | `str \| null` | ❌ | Optional module context (e.g., "recruitment") |
| `endpoint_mode` | `"assistant" \| "query"` | ✅ | Which API endpoint to test |
| `question` | `str` | ✅ | The user's question |
| `history` | `list[{role, text}]` | ❌ | Conversation history for follow-up context |
| `expected_answer_type` | `str` | ✅ | See table below |
| `expected_source_document_ids` | `list[int]` | ❌ | Document IDs that SHOULD appear in results |
| `expected_keywords` | `list[str]` | ❌ | Keywords that SHOULD appear in the answer |
| `forbidden_keywords` | `list[str]` | ❌ | Keywords that MUST NOT appear in the answer |
| `criticality` | `"low" \| "medium" \| "high"` | ❌ | Importance of this case (default: medium) |
| `slice_tags` | `list[str]` | ❌ | Tags for slicing metrics (e.g., ["overview", "vietnamese"]) |
| `notes` | `str \| null` | ❌ | Human notes about the case |

### Expected Answer Types

| Type | Description | Abstain Expected? |
|---|---|---|
| `overview` | Summary / high-level questions | No |
| `specific` | Detailed questions about procedures, rules | No |
| `follow_up` | Follow-up questions that need conversation history | No |
| `abstain` | Out-of-scope questions (no relevant documents) | **Yes** |
| `ambiguous` | Vague questions that lack context | Acceptable |
| `citation_required` | Questions that MUST include document citations | No |
| `general` | Greetings, thanks, generic questions | No |
| `compare` | Comparison questions between items | No |
| `no_answer` | Legacy alias for `abstain` | **Yes** |

### Endpoint Mode

- **`assistant`** — Tests the full pipeline via `/api/v1/assistant/respond`
  (retrieval + LLM answer + citations).
- **`query`** — Tests the retrieval pipeline only via `/api/v1/query`
  (no answer generation, only document ranking).

## Adding New Cases

### Principles

1. **Always set `tenant_id`** — Cases without a tenant_id will be rejected.
2. **Use realistic questions** — Synthetic/contrived questions give misleading metrics.
3. **Use real `expected_source_document_ids`** — Point to actual document IDs in your tenant's corpus.
4. **Cover diverse slices** — Include overview, specific, follow_up, abstain, citation_required.
5. **Use `forbidden_keywords`** — Especially for abstain cases, to detect hallucination.
6. **Set `criticality`** — "high" for cases that represent critical business logic.

### Per-Tenant Isolation

- Each case has exactly one `tenant_id`.
- The eval runner filters by tenant when requested.
- Metrics are **never mixed** across tenants.
- When adding cases for a new tenant, create cases with that tenant's ID and
  reference documents from that tenant's corpus.

### Example: Adding a new tenant

```jsonl
{"case_id": "acme_specific_001", "tenant_id": "acme_corp", "endpoint_mode": "assistant", "question": "What is the PTO policy?", "expected_answer_type": "specific", "expected_source_document_ids": [42, 43], "expected_keywords": ["PTO", "vacation", "days"], "forbidden_keywords": [], "criticality": "high", "slice_tags": ["specific", "english", "hr_policy"]}
```

## Running Evaluations

### API Mode (against running server)

```bash
# Start the server first
python -m uvicorn app.main:app --host 0.0.0.0 --port 8000

# Run all assistant cases for tenant "demo"
python scripts/run_phase12a_eval.py --mode assistant --tenant demo

# Run all query cases for tenant "demo"
python scripts/run_phase12a_eval.py --mode query --tenant demo

# Run a single case
python scripts/run_phase12a_eval.py --case-id demo_overview_001

# Run cases with a specific slice tag
python scripts/run_phase12a_eval.py --slice overview

# Limit number of cases
python scripts/run_phase12a_eval.py --limit 5

# Custom dataset
python scripts/run_phase12a_eval.py --dataset evals/phase1/golden_set.jsonl
```

### Direct Mode (in-process, no server needed)

```bash
python scripts/run_phase12a_eval.py --exec-mode direct --tenant demo
```

### Validate Dataset Only (CI-friendly)

```bash
python scripts/run_phase12a_eval.py --validate-only
```

## Metrics

### Retrieval Metrics

| Metric | Description |
|---|---|
| `retrieval_hit_at_1` | Expected document found in top-1 result |
| `retrieval_hit_at_3` | Expected document found in top-3 results |
| `retrieval_hit_at_5` | Expected document found in top-5 results |
| `retrieval_recall_at_5` | Fraction of expected docs found in top-5 |
| `retrieval_mrr` | Mean Reciprocal Rank of first expected doc hit |

### Answer Metrics

| Metric | Description |
|---|---|
| `has_answer_rate` | % of cases that produced a meaningful answer (>20 chars) |
| `has_citations_rate` | % of cases that included citations |
| `citation_doc_ids_valid_rate` | % of cases where all citations have valid doc IDs |
| `citation_same_tenant_rate` | % of cases where citations don't leak across tenants |
| `keyword_coverage_avg` | Average fraction of expected keywords found in answer |
| `forbidden_keyword_violation_rate` | % of cases containing forbidden keywords |
| `abstention_behavior_accuracy` | % of cases with correct abstain/answer behavior |

### Reporting Slices

All metrics are computed for:
- **Overall** — all evaluated cases
- **By tenant** — grouped by `tenant_id`
- **By slice tag** — grouped by `slice_tags`
- **By expected answer type** — grouped by `expected_answer_type`

> There is **no single "magic score"** in 1.2A. Each metric is reported individually.

## Programmatic Graders

All graders in `app/evals/graders/programmatic.py` are **pure deterministic functions**:

### Retrieval Graders
1. `retrieved_expected_doc(result_doc_ids, expected_doc_ids)` → (bool, count)
2. `hit_at_k(result_doc_ids, expected_doc_ids, k)` → bool
3. `recall_at_k(result_doc_ids, expected_doc_ids, k)` → float
4. `mrr(result_doc_ids, expected_doc_ids)` → float

### Answer Graders
5. `has_answer(answer_text)` → bool
6. `has_citations(answer_result)` → bool
7. `citation_doc_ids_exist(answer_result)` → bool
8. `citations_same_tenant(answer_result, tenant_id)` → bool
9. `keyword_coverage(answer_text, expected_keywords)` → (float, found, missing)
10. `forbidden_keyword_violation(answer_text, forbidden_keywords)` → (bool, violations)
11. `abstention_detected(answer_text)` → bool
12. `abstention_behavior_basic(answer_text, expected_answer_type)` → bool
